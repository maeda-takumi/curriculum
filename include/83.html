第1章：まず「思考法」を変える
──コードを書くのではなく「設計して翻訳させる」
1-1. ノーコード制作の本質的な意味
「ノーコード制作」という言葉は、しばしば誤解されます。
「コードを使わずにWebページを作る」という意味ではありません。むしろ正確には「自分でコードを書かずに、AIにコードを生成させてWebページを作る」です。この違いは非常に重要です。
完成したHTMLファイルにはコードがあります。
ブラウザはそのコードを読んでページを表示します。
あなたが変えるのは「誰がそのコードを書くか」です。エンジニアが書いていたコードを、Claudeが生成する。あなたの役割は「コードを書く人」から「何を作るかを設計して、Claudeに指示する人」へと移行します。この役割転換がノーコード制作の本質です。
そしてこの「設計して指示する」という役割は、ライターにとって最も得意な仕事と構造的に同じです。クライアントの要望を理解し、ターゲットへの訴求を設計し、それを正確な言語で指示として表現する。あなたはフェーズ0〜7でまさにこの能力を磨いてきました。
「コードを書けないことは、ノーコード制作においては弱点ではない。むしろ「設計と指示」に集中できるという強みになる。」
1-2. HTMLを「見る目」と「書く技術」は全く別物
「HTMLがわからないからWeb制作はできない」という思い込みは、2つの全く異なるスキルを混同しています。
HTMLを「書く技術」とHTMLを「読んで理解できる目」は別物です。
ノーコード制作に必要なのは後者だけです。そしてHTMLを「読む」ことは、ライターが文章を読むのと似ています。どんな情報が、どんな構造で、どの順番で並んでいるかを把握する力です。
Claudeが生成したHTMLを見たとき、コードの細部がわからなくても「このブロックがFVセクションで、この部分がCTAボタンで、この文字がキャッチコピーだ」という構造的な理解ができれば十分です。
修正指示も「このキャッチコピーを変えて」「このボタンの色を変えて」と日本語で言うだけです。HTMLのどの行を変えればいいかはClaudeが判断します。
上の図はClaudeによるHTMLノーコード制作の基本ワークフローです。
①設計→②指示→③生成→④確認→⑤修正という5ステップで、修正は②の指示ステップに戻る「ループ構造」になっています。
このループを回すたびに成果物の品質が上がります。重要なのは、どのステップにおいても「HTMLを自分で書く」というプロセスがないことです。
第2章：最低限知っておくべきHTML・CSSの「語彙」
2-1. 「知っておくと指示精度が上がる」最低限の知識
上の図は「知っておくべき知識」と「Claudeに任せていい部分」の境界線を示しています。
左側の青いゾーンの知識を持っていれば、ほぼすべてのLP・HP制作案件に対応できます。右側の紫のゾーンはClaudeに任せれば良く、自分で学ぶ必要はありません。
重要なのは、左側のリストはHTMLのタグ名と「その役割の大意」を知っていれば十分だということです。正確な構文を暗記する必要はまったくありません。
「h1は大きな見出し」「sectionはページのひとまとまり」「buttonはボタン」という程度の理解があれば、Claudeへの指示の中でこれらの言葉を自然に使えるようになります。

2-3. 「Tailwind CSS」を知っておくと指示がさらに楽になる
ClaudeにHTMLを生成させるとき、「Tailwind CSS（テイルウィンド CSS）を使って」という一言を指示に加えることを強く推奨します。
Tailwind CSSとは、HTMLの中に短いクラス名を書くだけでデザインを適用できるCSSフレームワークです。「tailwind CSSを使ってほしい」と伝えるだけで、Claudeが生成するHTMLのデザインが一段と洗練され、かつレスポンシブ対応も自動的に組み込まれます。
使うためのセットアップも簡単です。
Claudeが生成するHTMLの中に
<script src="https://cdn.tailwindcss.com"></script>という1行が自動で追加され、それだけで使えるようになります。
自分でインストールする必要はありません。「Tailwind CSSを使う」という指示を覚えておくだけで、生成されるHTMLの品質が大きく向上します。
📝Tailwind CSSを使ったClaudeへの指示の基本形
「以下の仕様でLPのHTMLを生成してください。

  ・モバイルファーストのレスポンシブ対応

  【デザイン仕様】
  ・メインカラー：#1B6EC2

  【LP構成】...(8-3 第4章以降で詳しく解説)」
→ 「Tailwind CSS（CDN使用）」の一言で、別途CSSファイルの作成が不要になる
→ Claudeがデザインの細かい調整もTailwindクラスで一括で管理してくれる
第3章：LPをブロックで設計する
──構造を理解してから指示する
3-1. LPは「感情設計のブロック」の積み重ね
左の図はLPのHTML構造をブロックで表したものです。
<header>・<section>・<footer>という要素がページの縦方向に積み重なっています。
LPとはこのブロックの並び順と各ブロックの内容で構成されています。この「ブロックの設計」がClaudeへの指示の出発点です。
右のグラフは指示の詳細度と修正回数の関係を示しています。
曖昧な1行指示では平均8.2回の修正が必要なのに対して、ライター式フルスペック指示では平均0.8回まで削減されています。
「詳細な指示を出すことが結果的に最速」というのが、このグラフの核心メッセージです。
LPのブロック設計を先に紙に書き出してからClaudeに指示すると、指示が自然と詳細になります。
「このLPは①FV→②問題提起→③解決策→④特徴3点→⑤CTA→⑥FAQ→⑦最終CTA」という構成が決まってから指示することで、Claudeは「全体の流れを理解した上でHTML全体を設計する」ことができます。
構成を決めずに指示すると、Claudeは「一般的なLPのテンプレート」を生成するだけになりがちです。
3-2. セクションごとの「役割」をClaudeに伝える重要性
LP制作でよく起きる問題の1つが「各セクションがバラバラな印象になる」ことです。
これはClaudeに「LPのHTMLを生成して」という指示だけを出したとき、Claudeが各セクションを独立した要素として生成するためです。
感情設計の「流れ」が伝わっていないと、各セクションがつながらないLPになります。
この問題を解決するのが「各セクションの役割と感情目標を指示に含める」ことです。
「問題提起セクションでは読者に「そうそう！」という強い共感を感じさせることを目標に」「解決策セクションでは「自分にもできそう」という希望を喚起することを目標に」という指示が、各セクションのコピーとデザインに反映されます。

第4章：「一発で高品質なHTML」を生成するフルスペック指示の書き方

この5層はそれぞれ独立した役割を持ち、すべてが揃ったとき一発で高品質なHTMLが生成されます。

第5章：生成されたHTMLを「ブラウザで確認する」方法
5-1. 生成されたコードをファイルに保存する手順
Claudeが生成したHTMLは、Claude.aiの画面上に長いテキストとして表示されます。これをブラウザで確認するためには、まずファイルに保存する必要があります。以下の手順で進めてください。
📋HTMLをファイルに保存する手順（Windows・Mac共通）
【STEP 1】Claudeが生成したHTMLコードをすべてコピーする
→ 生成されたコードの先頭（<!DOCTYPE html>）から末尾（</html>）まで全選択
→ コードブロックの右上の「コピー」ボタンを使うと便利
【STEP 2】テキストエディタを開く
→ Windowsなら「メモ帳」（無料・最初から入っている）
→ Macなら「テキストエディット」（無料・最初から入っている）
→ より快適に使いたい場合はVSCode（無料・インストール推奨）
【STEP 3】コードを貼り付けて保存
→ テキストエディタに貼り付ける
→ 「名前を付けて保存」で保存
→ ファイル名は「index.html」にする（拡張子が.htmlなら名前は何でもOK）
【STEP 4】ブラウザで開く
→ 保存したファイルをダブルクリックするだけ
→ Google Chrome・Safari・Edgeいずれでも確認できる
→ これだけでブラウザにページが表示される
→ 「なんかうまく表示されない」場合は次のセクションの対処法を確認
5-2. 表示の確認で見るべき3つのポイント
ブラウザで表示を確認するとき、いきなり細部を見るのではなく、以下の3つを順番に確認する習慣をつけてください。
この順番が、修正作業の効率を大きく上げます。
確認の順番
確認すること
問題があった場合のClaudeへの修正指示例
①全体の構成確認
ページをスクロールして、すべてのセクションが意図した順番で存在しているか
「FAQセクションがありません。CTAの上に追加してください」
②スマホ表示の確認
ブラウザのデベロッパーツール（F12）でスマホサイズに切り替えて、崩れていないか
「スマホ表示でFVのキャッチコピーが2行になって見切れています。文字サイズを小さくして」
③デザインの印象確認
ブランドボイスGLで設定したトーンと一致した雰囲気になっているか
「全体的に硬い印象です。もう少し温かみのある色調に調整してください」
5-3. デベロッパーツールの「スマホ確認」だけ覚える
HTMLの確認で最も重要な操作は「スマホ表示の確認」です。
LP・HPへのアクセスの60〜70%はスマホからです。PCで見た目が良くてもスマホで崩れているとクライアントから即修正依頼が来ます。スマホ表示の確認方法だけは必ず覚えてください。
💡スマホ表示の確認方法（Chrome推奨）
【操作手順】
1. Chromeでページを開いた状態でF12キーを押す（デベロッパーツールが開く）
2. 上部のアイコンの中から「スマホとタブレットのアイコン」（Toggle Device Toolbar）をクリック
3. 画面上部のドロップダウンで「iPhone SE」または「iPhone 14 Pro」を選ぶ
4. ページがスマホサイズで表示される。スクロールして全体を確認
【Macの場合】
→ F12の代わりにCommand ＋ Option ＋ Iでデベロッパーツールが開く
【確認するポイント】
・文字がはみ出していないか
・ボタンが小さすぎて押しにくくないか
・画像が正しいサイズで表示されているか
・セクションの余白が狭すぎてゴチャゴチャしていないか
→ 問題があればClaudeに「スマホで〇〇が〇〇になっています。修正してください」と伝えるだけ

第7章：よくある「うまく生成されないケース」と対処法
7-1. 「途中で出力が止まる」問題
LPのHTMLは非常に長いコードになるため、Claudeが「コンテキストの上限」に達して出力が途中で止まることがあります。
これは特にLP全体を一度に生成しようとするときに起きやすい問題です。

出力が途中で止まったときの対処法
【対処法①】「続きを書いてください」と送る
→ Claudeが途中から続きを出力してくれる
→ 繰り返し使える。3〜5回送れば完成することが多い
【対処法②】LP全体を一度に生成させず、セクションごとに分割して生成する
→ 「まずヘッダーとFVだけ生成してください」→確認→「次に問題提起セクションを追加して」
→ 分割することで各セクションの品質が上がる副効果もある
【対処法③】コピーをプレースホルダーにして、まず構造だけを生成する
→ テキストが短くなるのでコード全体が短縮される
→ 構造が完成してから、コピーをあとから差し込む
7-2. 「スマホで表示が崩れる」問題
Tailwind CSSを指定しても、スマホ表示が崩れることがあります。
特に「横並びにしたカードがスマホで横にはみ出す」という問題が頻発します。これはレスポンシブ設定の漏れが原因です。
💡スマホ崩れを防ぐClaudeへの追加指示
初回の指示に以下を追加するだけで、スマホ崩れが大幅に減ります：
「スマホ表示を最優先にしてください。
具体的には：
・横並びのカード・グリッドはスマホでは必ず縦1列に変わること
・画像はスマホで画面幅いっぱいに表示されること
・文字サイズはスマホで読みやすい大きさを保つこと
・横スクロールが発生しないこと（overflow-x: hiddenを設定）
Tailwindのresizing（sm:・md:・lg:）を適切に使ってください」
7-3. 「Tailwind CSSが適用されない」問題
「Tailwind CSSを使って生成してもらったのに、見た目が普通のHTMLと変わらない」という問題が起きることがあります。多くの場合、HTMLのhead部分にTailwindのCDNリンクが入っていないか、リンクが誤っているのが原因です。

Tailwind CSSが効いていないときの確認と修正
【確認方法】
ClaudeのHTMLの<head>タグの中に以下の行があるか確認してください：
  <script src="https://cdn.tailwindcss.com"></script>
【修正指示】
「Tailwind CSSのスタイルが適用されていないようです。
  <head>内にTailwindのCDNリンクが正しく入っているか確認して、
  もし入っていなければ追加してください」
【代替案】
最初の指示で明示的に指定することで予防できます：
「<head>内に必ず以下を含めてください：
  <script src="https://cdn.tailwindcss.com"></script>
  <link href="https://fonts.googleapis.com/css2?family=Noto+Sans+JP:wght@400;700&display=swap" rel="stylesheet">」
7-4. 「コピーのトーンがブランドボイスと合わない」問題
BV-GLを指定したのに生成されたコピーがブランドボイスと合わない場合、多くはBV-GLの情報量が足りていないことが原因です。
特に「3つの形容詞」だけを指定して、「それぞれの具体的な意味」「使うべき言葉のリスト」「避けるべき言葉のリスト」が省略されている場合に起きやすい問題です。

💡BV-GLの指定を「もれなく」含めるチェックリスト
Claudeへの指示に以下がすべて入っているか確認してください：
✅ ブランドパーソナリティ：形容詞3つ＋それぞれの意味
✅ トーン：文末表現・一文の長さ・読者への呼びかけ方
✅ 積極的に使う言葉のリスト（10語以上）
✅ 絶対に使わない言葉のリスト（10語以上）
✅ サンプル文（「このような文体で書いてください：[50字の例文]」）
→ サンプル文を提示するのが最も効果的なBVGL指定方法
→ Claudeはサンプル文のリズム・語彙・温度感を参照して文章を生成できる
7-5. 「日本語フォントが反映されない」問題
日本語フォントを指定したのに、ブラウザで確認するとシステムデフォルトのフォントになっているという問題があります。これはGoogle Fontsのリンクが正しく設定されていないか、Tailwindの設定でフォントが上書きされていないことが原因です。
📝日本語フォント（Noto Sans JP）を確実に適用する指示テンプレート
「以下のフォント設定を確実に実装してください。
【headに追加するリンク】
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
  <link href="https://fonts.googleapis.com/css2?family=Noto+Sans+JP:wght@400;500;700&display=swap" rel="stylesheet">
【Tailwindのフォント設定】
  tailwind.config の fontFamily に Noto Sans JP を設定するか、
  bodyに直接 style属性でNoto Sans JPを指定してください。
サイト全体のデフォルトフォントをNoto Sans JPにしてください」

第9章：「コードを読む目」を育てる──最低限の確認スキル
9-1. HTMLを「音読する」という習慣
HTMLを理解するための最も簡単なアプローチは「音読する」ことです。
<section id="fv">を見たら「FVセクション開始」、</section>を見たら「FVセクション終了」、<h1 class="text-4xl font-bold">を見たら「大きな太字の見出し開始」と頭の中で読む。この習慣が自然についてくると、Claudeが生成したHTMLのどこに何があるかが一目でわかるようになります。
完全な理解は不要です。
「だいたいこのあたりがFVで、このあたりがCTA」という構造的な把握ができれば、「FVのキャッチコピーを変えたい」という修正指示を正確に出せます。
Claudeへの修正指示の精度は、この「構造的な把握」があるかどうかで大きく変わります。
9-2. よく見かける「Tailwind CSSのクラス名」と意味の対応
Claudeが生成するHTMLにはTailwindのクラス名が大量に使われます。
すべてを覚える必要はまったくありませんが、「見てだいたいの意味がわかる」クラスを10〜20個知っておくと、修正指示の精度が上がります。
Tailwindクラス名
意味
修正指示への活用例
text-4xl / text-6xl
文字サイズ（4xlが大・6xlがより大）
「FVのキャッチのtext-4xlをtext-5xlに変えて」
font-bold / font-normal
太字・通常の太さ
「この見出しにfont-boldを追加して」
text-white / text-gray-700
文字色（白・グレー）
「ボタンの文字をtext-whiteに変えて」
bg-blue-600 / bg-white
背景色（青・白）
「このセクションのbgをbg-gray-50に変えて」
py-20 / px-8
縦・横の余白（paddingの略）
「このセクションのpy-20をpy-32にして余白を広げて」
flex / grid
横並びレイアウト
「このカードをgrid grid-cols-3にして3列にして」
rounded-full / rounded-lg
ボタン等の角丸（全丸・中程度）
「CTAボタンをrounded-lgからrounded-fullに変えて」
shadow-md / shadow-xl
影（中程度・大きめ）
「カードにshadow-mdを追加して立体感を出して」
sm: / md: / lg:
レスポンシブ（スマホ/タブレット/PC）
「md:grid-cols-3をmd:grid-cols-2にして」
hidden / block
非表示・表示
「スマホでこの要素をhiddenにしてPCだけ表示して」
このテーブルはClaudeへの修正指示を出すときの「辞書」として使ってください。
完全に暗記する必要はなく、「こういうクラスがあるんだな」という感覚を持つだけで十分です。
実際の制作では「Claudeにどう指示するか」を考えることに集中して、Tailwindの詳細はClaudeに任せてください。
